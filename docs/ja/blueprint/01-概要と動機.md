# 第1章 概要と動機

## 1.1 データトンネルプロトコルとは

データトンネルプロトコル（Data Tunnel Protocol、略称 DTP）は、iFay エコシステムにおける6大プロトコルの一つであり、**ネゴシエーション型データ伝送チャネルプロトコル**です。端末デバイスと Fay 間の双方向データ収集・注入を担当します。

DTP はアプリケーション層プロトコルとして、既存のトランスポートプロトコル（BLE、RTSP、WebSocket、TCP など）の上に構築されます。下位層の伝送方法には関与せず、「何を伝送するか、どう組織するか、どうネゴシエーションするか、どう保障するか」のみを定義します。

## 1.2 プロトコルの動機：データ主権

従来のモデルでは、アプリケーションがユーザーの行動データを独自に収集してレコメンデーション等に利用し、データはプラットフォームに帰属していました。ユーザーは自分のデータに対する制御権を持たず、どのデータを誰が使用できるかを決定できませんでした。

DTP の核心的価値提案は**データ主権**です。AI 時代において、個人データは個人に帰属すべきであり（iFay を通じて Personal Data Heap に管理委託）、各アプリケーション開発者に散在すべきではありません。

DTP モデルにおけるデータフロー：

1. すべての端末データは DTP を通じて iFay の Personal Data Heap に収集される
2. 端末アプリケーションがパーソナライズデータを必要とする場合、iFay に申請する
3. iFay は人間のように、どの情報を提供するか、どの程度提供するかを判断し、フィルタリングされた最小限のデータセットを返す
4. データ主権は常にユーザー（ヒューマンプライム）の手にある

## 1.3 2つのコアデータフロー

DTP は2つのコアデータフローを実現します：

- **データ収集（Terminal → Fay）**：端末で生成されたデータを iFay の Personal Data Heap に永続化保存し、データ管理を実現
- **データ注入（Fay → Terminal）**：iFay がフィルタリングと判断を経た最小限のデータセットを端末アプリケーションに一時的に提供し、プライバシーを漏洩せずにパーソナライズサービスを実現

## 1.4 コンテキスト化データ

データは元の場面から切り離されると意味を失う可能性があります。例えば：

- ユーザーがデリバリーアプリで冷凍緑豆スープを注文し、同時にその日の気温が32℃と記録されていれば、ユーザーが暑さのために冷たい飲み物を選んだことを示す
- 気温が12℃であれば、ユーザーが冷たい飲み物を好むことを示す

DTP はプロトコルレベルでコンテキストメタデータを携帯し、データ収集時にコンテキストを即座に復元できるようにします。各データフラグメント（Fragment）は、データタイプ、ソース識別子、収集環境などの情報を含む構造化されたコンテキストメタデータを携帯します。

## 1.5 CAP との連携

DTP は制御認可プロトコル（CAP）と連携して動作します：

- **CAP** は接続認可、身元認証、鍵交換を担当
- **DTP** は実際のネゴシエーション型データストリーム伝送を担当

両者は共に「クライアントの直接制御」能力を実現し、UI ベースのインタラクションを必要としません。DTP は CAP が身元認証と鍵交換を完了した後にデータ伝送を開始し、通信双方の身元が信頼でき、鍵が利用可能であることを保証します。
