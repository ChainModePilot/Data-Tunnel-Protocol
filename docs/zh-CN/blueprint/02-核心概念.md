# 第二章 核心概念

## 2.1 主从关系模型

DTP 具有明确的主从关系：

- **主端（Master）**：自然人用户或 Fay（iFay / coFay）——数据的最终归属方和决策方
- **从端（Slave）**：软件或硬件终端——数据的生产方或消费方

### 关键约束

| 约束 | 说明 | 示例 |
|------|------|------|
| 单一控制者 | 同一时刻，一个终端只能有一个 Fay "附体" | 用户的智能手表在某一时刻只能被用户自己的 iFay 控制 |
| 旁观者机制 | 控制者 Fay 可以邀请或授权其他 Fay 旁观（只读权限） | 用户的 iFay 控制家中的智能摄像头，同时邀请家庭医生的 coFay 旁观健康监测数据流 |
| 主端调阅权 | 主端有权调阅从端数据，从端多数情况不能拒绝 | iFay 要求企业笔记本提交浏览记录，笔记本的 DLP 代理因公司合规政策拒绝了该请求 |
| 从端申请制 | 从端向主端请求数据注入时，主端拥有完全决定权 | 打车软件向 iFay 申请用户的家庭和公司地址，iFay 判断用户正在通勤，仅提供公司地址 |
| 多主端复用 | 从端可能被多个主端在不同时段复用 | 家庭共享的智能音箱，白天由妈妈的 iFay 附体，晚上由爸爸的 iFay 附体 |

## 2.2 参与模式

DTP 支持两种参与模式：

- **控制者（Controller）**：当前"附体"在终端上的 Fay，拥有完全读写权限
- **旁观者（Observer）**：被控制者邀请或授权的其他 Fay，仅拥有只读权限

旁观者仅能接收数据流的只读副本，不能发起请求或修改约定。

## 2.3 约定（Agreement）

约定是主端与从端之间协商达成的数据传输契约，定义了数据传输的所有参数：

- **数据类型/范围**：传输什么数据
- **传输模式**：一次性（one_time）、周期性（periodic）、流式（streaming）
- **传输频率**：数据发送的频率
- **有效期**：约定的有效时长
- **优先级**：低（low）、正常（normal）、高（high）、紧急（critical）

所有数据传输必须基于双方协商达成的约定，无"裸传输"。

## 2.4 数据片段（Fragment）

Fragment 是 DTP 中的数据单元，具有以下特征：

- **全局唯一标识符**（Fragment_ID）
- **原始时间戳**（Origin_Timestamp）：数据实际产生的时刻，非传输时刻
- **DAG 依赖关系**：与其他 Fragment 的关联
- **约定归属**：通过 Agreement_ID 表明所属约定
- **上下文元数据**：结构化的语境信息

## 2.5 有向无环图（DAG）依赖

Fragment 之间通过 DAG 边表达依赖关系，支持三种关系类型：

| 关系类型 | 含义 | 示例 |
|---------|------|------|
| `derived_from` | 派生自 | "每日步数汇总"片段派生自当天的各个步数记录片段 |
| `annotates` | 注释 | 天气数据片段注释外卖订单片段，解释用户为何在高温下点了冰饮 |
| `supersedes` | 替代 | 用户更新收货地址后，新地址片段替代旧地址片段 |

DAG 结构确保数据在归集时就建立关联，帮助 iFay 理解数据的演化脉络和因果关系。

## 2.6 术语表

| 术语 | 定义 |
|------|------|
| iFay | Individual Fay，绑定到特定自然人（人类原型）的个人 AI 化身（数字孪生） |
| coFay | Common Fay，公共角色 AI（类似 Agent） |
| Fay | 拟人化 AI 代理的通用术语 |
| FayGer | Fay 的容器/运行时环境（类似 Docker/JRE），属于"公共场所"，不应访问明文数据 |
| 人类原型（Human Prime） | iFay 所绑定的自然人 |
| Faying | iFay 与其人类原型处于连接/配对状态 |
| Personal Data Heap | iFay 的私有数据管理模块，存储多种格式的数据（人类原型的"日记"） |
| Sensor | 基于 CAP + DTP 构建的 iFay "神经系统"，接收数据流 |
| Device Driver Hub | 集成设备驱动的驱动中心层 |
| DTP_Engine | DTP 协议的核心处理引擎，负责帧的编码、解码、加密、解密及传输管理 |
